Built on what already works

Crucible12 doesn't reinvent the agent loop — it pairs OpenCode (the scripts below) or a built-from-scratch chat UI (the desktop app) with a llama.cpp backend, and a model picked to fit whatever you're running it on. The project started as a build tuned for one specific high-end machine — that tier still exists, it's just no longer the only one:

Don't have that? The desktop app and the tiers/ scripts cover everything from a no-GPU laptop (3B, CPU-only) up through 4-6GB, 6-8GB, 10-12GB, and 16GB-VRAM cards before reaching the big-rig presets below.

Prefer the command line? (Windows, the original big-rig presets)

The PowerShell scripts this project started as — for the four MoE presets that need serious VRAM+RAM. Choose one and the specs, memory split, and exact setup commands update live. For everything below 16GB VRAM, see tiers/ or just use the desktop app above.

Defaults are sensible starting points. Lower --n-cpu-moe until VRAM sits ~30GB; raise it if you hit CUDA out-of-memory.

💡 Enable DDR5 EXPO/XMP in your BIOS before benchmarking. The hybrid presets stream experts from system RAM every token, so generation speed is bandwidth-bound — EXPO is worth up to ~3× throughput.

Why active params, not total size, decide speed here

Once MoE experts spill into system RAM, generation speed tracks the active parameters per token (and RAM bandwidth), not the total model size. That's why an 80B model with 3B active beats a 235B model with 22B active on this box — and why the genuinely huge models are loadable curiosities, not daily drivers.
